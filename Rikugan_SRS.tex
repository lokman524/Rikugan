\section{Introduction}

\subsection{Purpose}
This Software Requirements Specification (SRS) document specifies the requirements for Rikugan, version 1.0. The system is designed as a bounty-based project management application that combines traditional Kanban board functionality with gamified task assignment and reward mechanisms. This SRS covers the complete web-based application including user authentication, task management, role-based access control, and reward tracking systems.

\subsection{Document Conventions}
This document follows the IEEE 830-1998 standard for software requirements specifications. The following typographical conventions are used:

\begin{itemize}
    \item \textbf{Bold text} indicates system components, user interface elements, and important terms
    \item \textit{Italic text} represents user actions and system responses
    \item \texttt{Monospace text} denotes code elements, database fields, and technical specifications
    \item Requirements are numbered using the format REQ-XX for functional requirements and NFR-XX for non-functional requirements
    \item Priority levels are classified as High (essential for system operation), Medium (important but not critical), and Low (nice-to-have features)
\end{itemize}

\subsection{Intended Audience and Reading Suggestions}
This SRS is intended for the following audiences:

\begin{itemize}
    \item \textbf{Developers and Software Engineers}: Should focus on Section \ref{sec:system-features} (System Features) and Section \ref{sec:external-interfaces} (External Interface Requirements) for implementation details
    \item \textbf{Project Managers}: Should review Section \ref{sec:overall-description} (Overall Description) and Section \ref{sec:other-requirements} (Other Requirements) for project scope and constraints
    \item \textbf{Quality Assurance Testers}: Should concentrate on Section \ref{sec:system-features} for functional requirements and Section \ref{sec:nonfunctional} for non-functional requirements
    \item \textbf{End Users}: Should review Section \ref{sec:overall-description} for system overview and user role descriptions
    \item \textbf{Course Instructors}: Should examine all sections to evaluate project completeness and feasibility
\end{itemize}

The document is organized to provide a high-level overview first, followed by detailed technical specifications. Readers are recommended to start with Section \ref{sec:overall-description} (Overall Description) before proceeding to their area-specific sections.

\subsection{Product Scope}
Rikugan is a web-based application designed to revolutionize project management through gamification and role-based task assignment. The system addresses the need for better organization and resource allocation in software development projects by implementing a bounty-based reward system.

\textbf{Key Benefits and Objectives:}
\begin{itemize}
    \item Improved task visibility and progress tracking through Kanban-style boards
    \item Increased developer motivation through gamified bounty rewards
    \item Clear role-based hierarchies with appropriate access controls
    \item Streamlined project communication and collaboration
    \item Efficient resource allocation and deadline management
\end{itemize}

The system supports three distinct user roles (Goons, Hashira, and Oyakatasama) with escalating privileges and responsibilities, creating a structured environment that promotes both individual accountability and team collaboration.

\section{Overall Description}
\label{sec:overall-description}

\subsection{Product Perspective}
Rikugan is a new, self-contained web-based application designed specifically for project management in software development environments. The system draws inspiration from popular task management tools like Trello and Jira but incorporates unique gamification elements through a bounty-based reward system.

\begin{itemize}

\end{itemize}

\section{System Features}
\label{sec:system-features}

\subsection{User Authentication and Authorization}

\subsubsection{Description and Priority}
\textbf{Priority: High} - Essential for system security and role-based access control

This feature provides secure user registration, login, and role-based access control for the three user types: Goons, Hashira, and Oyakatasama. The system implements team-based license validation to control task creation limits.

\subsubsection{Stimulus/Response Sequences}
\begin{enumerate}
    \item User attempts to access the system → System redirects to login page if not authenticated
    \item User enters credentials → System validates and grants role-appropriate access
    \item Admin creates new user account → System generates credentials and assigns role
    \item User attempts unauthorized action → System denies access and logs attempt
\end{enumerate}

\subsubsection{Functional Requirements}

\paragraph{REQ-1: User Registration}
The system shall allow administrators (Oyakatasama) to create new user accounts with the following information:
\begin{itemize}
    \item Username (unique, 3-50 characters)
    \item Email address (valid format)
    \item Initial password (minimum 8 characters)
    \item Assigned role (Goon, Hashira, or Oyakatasama)
\end{itemize}

\paragraph{REQ-2: User Login}
The system shall authenticate users using username/email and password combination and maintain session state for 8 hours of inactivity.

\paragraph{REQ-3: Role-Based Access Control}
The system shall enforce the following access levels:
\begin{itemize}
    \item \textbf{Goons}: Task selection, status updates, profile viewing
    \item \textbf{Hashira}: All Goon functions plus task creation and team monitoring
    \item \textbf{Oyakatasama}: All functions plus user management and system administration
\end{itemize}

\paragraph{REQ-4: License Management}
The system shall implement a team-based license system where one hard-coded license key is configured per team deployment. Without a valid license, the entire project shall be inaccessible. The license is managed through environment configuration, not through the admin interface.

\subsection{Task Management System}

\subsubsection{Description and Priority}
\textbf{Priority: High} - Core functionality for project management

This feature implements the bounty-based task management system with Kanban-style boards, task assignment, and progress tracking.

\subsubsection{Stimulus/Response Sequences}
\begin{enumerate}
    \item Hashira creates new task with bounty → System adds task to available tasks pool
    \item Goon selects task → System assigns task and moves to "In Progress" status
    \item User updates task status → System updates Kanban board and notifies stakeholders
    \item Task reaches deadline → System applies penalties if incomplete
\end{enumerate}

\subsubsection{Functional Requirements}

\paragraph{REQ-5: Task Creation}
The system shall allow Hashira to create tasks with the following attributes:
\begin{itemize}
    \item Task name (required, max 100 characters)
    \item Description (required, max 1000 characters)
    \item Bounty amount (required, positive number)
    \item Deadline (required, future date)
    \item Priority level (High, Medium, Low)
    \item Required skills/tags
\end{itemize}

\paragraph{REQ-6: Task Assignment}
The system shall allow Goons to select and join available tasks, with automatic assignment to the first qualified user who selects the task.

\paragraph{REQ-7: Kanban Board}
The system shall display tasks in a Kanban board with the following columns:
\begin{itemize}
    \item Available
    \item In Progress
    \item Review
    \item Completed
\end{itemize}

\paragraph{REQ-8: Task Status Updates}
The system shall allow assigned users to update task status and add progress comments.

\paragraph{REQ-9: Deadline Management}
The system shall track task deadlines and apply monetary penalties for missed deadlines as defined by system configuration.

\paragraph{REQ-10: Task Deletion}
The system shall allow Hashira and Oyakatasama to delete tasks from the board. Tasks can only be deleted if they are in "Available" status or if no user is currently assigned to them.

\subsection{User Profile and Reward System}

\subsubsection{Description and Priority}
\textbf{Priority: Medium} - Important for gamification and user engagement

This feature manages user profiles, tracks earnings, and implements the reward/penalty system.

\subsubsection{Stimulus/Response Sequences}
\begin{enumerate}
    \item User completes task → System awards bounty to user's account
    \item User misses deadline → System applies penalty from user's account
    \item User views profile → System displays current balance and task history
    \item User achieves milestone → System displays achievement notification
\end{enumerate}

\subsubsection{Functional Requirements}

\paragraph{REQ-11: User Profile Management}
The system shall maintain user profiles containing:
\begin{itemize}
    \item Personal information (username, email, role)
    \item Current monetary balance
    \item Task completion history
    \item Performance statistics
\end{itemize}

\paragraph{REQ-12: Bounty Reward System}
The system shall automatically credit user accounts with bounty amounts upon successful task completion.

\paragraph{REQ-13: Penalty System}
The system shall apply configurable monetary penalties for missed deadlines or unsatisfactory task completion.

\paragraph{REQ-14: Performance Tracking}
The system shall track and display user performance metrics including:
\begin{itemize}
    \item Tasks completed
    \item Average completion time
    \item Success rate
    \item Total earnings
\end{itemize}

\subsection{Notification System}

\subsubsection{Description and Priority}
\textbf{Priority: Medium} - Important for user engagement and communication

This feature provides real-time notifications for task assignments, deadlines, and system events.

\subsubsection{Stimulus/Response Sequences}
\begin{enumerate}
    \item Task assigned to user → System sends notification to user
    \item Deadline approaching → System sends reminder notification
    \item Task status changed → System notifies relevant stakeholders
    \item User achieves milestone → System sends congratulatory notification
\end{enumerate}

\subsubsection{Functional Requirements}

\paragraph{REQ-15: Task Notifications}
The system shall notify users when:
\begin{itemize}
    \item New tasks are assigned to them
    \item Task deadlines are approaching (24 hours before)
    \item Task status is updated by team members
    \item Tasks are completed or cancelled
\end{itemize}

\paragraph{REQ-16: Achievement Notifications}
The system shall notify users of achievements such as:
\begin{itemize}
    \item First task completion
    \item Earning milestones
    \item Performance improvements
    \item Role promotions
\end{itemize}

\section{Other Non-Functional Requirements}
\label{sec:nonfunctional}

\subsection{Performance Requirements}

\subsubsection{Response Time Requirements}
\begin{itemize}
    \item Page loading: All pages must load within 3 seconds under normal network conditions
    \item API response time: 95\% of API calls must respond within 500ms
    \item Database queries: Complex queries must execute within 2 seconds
    \item User authentication: Login process must complete within 1 second
\end{itemize}

\subsubsection{Throughput Requirements}
\begin{itemize}
    \item Concurrent users: System must support up to 50 concurrent users (suitable for undergraduate project scope)
    \item Task operations: System must handle 100 task status updates per minute
    \item Notification delivery: Real-time notifications must be delivered within 5 seconds
\end{itemize}

\subsubsection{Scalability Requirements}
\begin{itemize}
    \item User capacity: System designed to handle up to 200 registered users
    \item Task capacity: Support for up to 1000 active tasks simultaneously
\end{itemize}

\subsubsection{Resource Usage}
\begin{itemize}
    \item Memory usage: Frontend application should use less than 100MB RAM per browser tab
    \item Storage efficiency: Database size should not exceed 1GB for typical usage
    \item CPU utilization: Server CPU usage should remain below 70\% under normal load
\end{itemize}

\subsection{Safety Requirements}

\subsubsection{User Safety Measures}
\begin{itemize}
    \item Prevent unauthorized access to sensitive user information
    \item Implement session timeout to protect unattended sessions
    \item Provide clear confirmation dialogs for destructive actions (task deletion, account removal)
    \item Sanitize all user inputs to prevent malicious code execution
\end{itemize}

\subsection{Security Requirements}

\subsubsection{Authentication \& Authorization}
\begin{itemize}
    \item Implement secure password policies (minimum 8 characters, complexity requirements)
    \item Use industry-standard password hashing (bcrypt with salt)
    \item Implement JWT-based authentication with token expiration
    \item Enforce role-based access control for all system functions
    \item Provide secure logout functionality that invalidates tokens
\end{itemize}

\subsubsection{Data Security}
\begin{itemize}
    \item Encrypt sensitive data at rest (user credentials, personal information)
    \item Use HTTPS for all client-server communications
    \item Implement SQL injection prevention through parameterized queries
    \item Sanitize all user inputs to prevent XSS attacks
    \item Validate and escape output data before rendering
\end{itemize}

\subsubsection{System Security}
\begin{itemize}
    \item Regular security updates for all dependencies
    \item Implement rate limiting to prevent brute force attacks
    \item Monitor and log security-relevant events
    \item Implement proper error handling that doesn't leak sensitive information
\end{itemize}

\subsubsection{Usability}

    \item Clear error messages with actionable guidance
    \item Responsive design supporting multiple screen sizes
    \item Accessibility compliance with WCAG 2.1 Level A standards
\end{itemize}

\subsubsection{Reliability}
\begin{itemize}

    \item Graceful degradation when external services are unavailable
    \item Automatic error recovery for transient failures
    \item Data consistency maintained across all operations
\end{itemize}

\subsubsection{Maintainability}
\begin{itemize}

\subsection{Business Rules}

\subsubsection{User Role Hierarchy}
\begin{itemize}
    \item Only Oyakatasama can create, modify, or delete user accounts
    \item Only Hashira and Oyakatasama can create new task bounties
    \item Only Goons and Hashira can select and work on tasks
    \item Users can only update status of tasks assigned to them
    \item Users cannot exceed their available balance when penalties are applied
\end{itemize}

\subsubsection{Task Management Rules}
\begin{itemize}
    \item Each task can only be assigned to one user at a time
    \item Task bounties must be positive monetary values
    \item Task deadlines must be future dates when created
    \item Completed tasks cannot be modified or reassigned
    \item Task status progression follows: Available → In Progress → Review → Completed
\end{itemize}

\subsubsection{Financial Rules}
\begin{itemize}
    \item Bounty payments are processed automatically upon task completion
    \item Deadline penalties are applied automatically when deadlines are missed
    \item Users cannot have negative account balances below -\$100
    \item All financial transactions must be logged with timestamps
    \item Only administrators can manually adjust user account balances
\end{itemize}

\subsubsection{License and Access Rules}
\begin{itemize}
    \item Each team deployment uses a single hard-coded license key
    \item License validation occurs at system startup
    \item Expired or invalid licenses prevent all access to the system
    \item License keys are configured through environment variables by developers, not managed through the admin interface
\end{itemize}

\subsubsection{Notification Rules}
\begin{itemize}
    \item Task deadline reminders sent 24 hours before due date
    \item Assignment notifications sent immediately upon task selection
    \item Achievement notifications sent upon reaching milestones
    \item Administrative notifications have higher priority than user notifications
\end{itemize}
